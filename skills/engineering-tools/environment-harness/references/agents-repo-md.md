# `AGENTS.repo.md` Spec

The repo-owned context file. Lives at `<repo-root>/AGENTS.repo.md`. Captures what is true of *this repo only*; everything else belongs in `AGENTS.<profile>.md`, the org or workspace `AGENTS.md`, or system-level config.

## File contract

| Property | Value |
|---|---|
| Path | `<repo-root>/AGENTS.repo.md` |
| Size | ‚Č§~100 lines |
| Tone | Imperative, declarative. No hedging. |
| Sections | Up to 10, numbered, from the scaffold below |
| Not applicable to this repo | Omit the section heading entirely (e.g. section 9 when the repo follows org PR defaults) |
| Applicable but answer unknown | Keep the heading; leave the body as `<!-- unknown ‚ÄĒ repo team to fill in -->` |
| Duplication | One canonical answer per topic. Link instead of restating. |
| Commit | Manual ‚ÄĒ print the draft, confirm, then write |

## Pre-flight detection

Scan the repo to seed each section. Surface findings as suggestions, not as committed content.

| Detect | Source |
|---|---|
| Primary language | `package.json`, `pyproject.toml`, `go.mod`, `composer.json`, `*.tf` |
| Sync-owned files | Grep for the central-sync header used by the organization |
| Generated paths | `.gitignore`; common codegen targets: `generated/`, `*.gen.ts`, `prisma/client`, `dist/`, OpenAPI types |
| Build / test commands | `Makefile`, `package.json#scripts`, `pyproject.toml#tool.poetry.scripts` |
| Devcontainer | Presence of `.devcontainer/` ‚ÄĒ surface in section 5 only if non-standard |
| Owner | `CODEOWNERS` |
| Ref + commit being analyzed | `git symbolic-ref --short -q HEAD || git describe --all --exact-match HEAD || printf 'detached'`, `git rev-parse --short HEAD` ‚ÄĒ record both in the file footer so detached checkouts have an anchor |

## Section scaffold

Defaults that apply to every section:

- **Don't duplicate lists that can be derived from code.** When a list is volatile (workflow files, generated paths, env vars, package versions), point at the source location (grep query, directory, header line) instead of enumerating.
- **Pull commands and configuration from real repo evidence** (`Makefile`, `package.json#scripts`, `pyproject.toml`, `justfile`, `Dockerfile`). Never invent commands or paths.
- **Mark unconfirmed scans with `<!-- verify -->`** for content the agent surfaced but the engineer did not directly confirm; the repo team removes the marker after a human read-through.
- **State the ref and commit** you analyzed in the file footer (see section menu).

### 1. What this repo is

2‚Äď3 sentences naming what this repo is and how it fits into the system. Use verb-and-object form, regardless of repo type (service, library, mobile app, Terraform module, internal tooling, etc.).

- Reject: README boilerplate, marketing language, vague phrases like *"core service for X"*, and descriptive metaphors with no technical content (*"serverless trash can"*). Anchor the sentence in concrete behavior ‚ÄĒ what input, what output, who calls it.
- Example: *"Returns customer billing contacts to the account service when a customer record changes."*

### 2. Owner / on-call

Squad name + GitHub handle of the on-call rotation. Seed from `CODEOWNERS` when present.

- Reject: an empty handle, a personal name without a handle, or generic answers like *"the team"*. Owner attribution is the field most often left blank; do not let it slip through. If `CODEOWNERS` is missing and the engineer can't name a concrete handle, mark this section `<!-- unknown ‚ÄĒ repo team to fill in -->` rather than fabricating.

### 3. Sync-owned files

Files in this repo that look local but are managed by a central template or sync repository.

Prefer pointing at the grep over enumerating every file:

```
All files containing the organization's central-sync header are sync-owned. Run the repo's documented grep for that header to list them.
```

Only call out individually the high-traffic exceptions an agent is most likely to edit incorrectly:

```
- `.github/workflows/deploy.yml` ‚ÄĒ sync-owned. Edit source at `<central-templates-repo>/workflow-templates/deploy.yml`.
```

### 4. Generated files

Paths the agent must not edit (codegen output, generated clients, OpenAPI types, protobuf stubs).

When many outputs come from a single source, link to the source rather than enumerating every output path:

```
- Everything under `src/types/openai.gen/*` is regenerated by `npm run gen:openai` from `openapi.yaml`. Edit `openapi.yaml`, not the generated files.
```

For a single generated file, the short form is fine: `` `<path>` ‚ÄĒ do not edit. Regenerated by `<command>`. ``

If a path *looks* generated but is actually hand-edited in this repo (e.g. database migration files that mix generated DDL with hand-written data migrations), do not list it here.

### 5. Repo-specific build / test / run

Anything an agent needs beyond the repo's standard bootstrap (e.g. `make devcontainer`, `pnpm install && pnpm test`, `terraform init`, `./gradlew build` ‚ÄĒ whatever this repo actually uses). Pre-bootstrap host steps. Long-startup timeouts. Required local fixtures. Skip the section if the standard bootstrap is sufficient on its own.

Every command listed must be traceable to the repo (`Makefile`, `package.json#scripts`, `pyproject.toml#tool.poetry.scripts`, `justfile`, `Dockerfile`, CI workflow). If the engineer dictates a command that isn't in the repo, ask where it lives ‚ÄĒ if they can't say, mark the bullet `<!-- verify -->` rather than recording it as fact.

### 6. Cross-repo dependencies

Upstream and downstream components this repo depends on or that depend on it ‚ÄĒ services it calls, libraries it consumes, Terraform modules it references, services that call it, packages that depend on it. Optional for v0. Format:

- **Depends on:** ...
- **Depended on by:** ...

### 7. Domain terms

Local jargon translation. Format: `` `<term>` = <translation> ``. Skip if no non-obvious terms exist; do not fabricate.

### 8. Common pitfalls

Concrete failure modes. Format: *"If `<symptom>`, check `<cause>` first, not `<wrong-cause>`."*

- Reject: generic statements (*"flaky tests sometimes"*). Each pitfall traces to one specific incident.

### 9. Repo-specific PR conventions

Only if this repo's PR conventions diverge from the org or workspace defaults. Skip when the repo follows the defaults ‚ÄĒ most repos do.

### 10. Pointers

Locations of `docs/decisions/`, deeper architecture docs, runbooks. Short link list.

## Tone rules

| Rule | Example |
|---|---|
| Imperative and declarative | *"Edit the source template in `<central-templates-repo>/...`. Do not edit `.github/workflows/deploy.yml` directly."* |
| No hedging | Reject *"you may want to consider..."*; rewrite as *"do X"* |
| One canonical answer per topic | Merge or link ‚ÄĒ never duplicate |
| Link to authoritative source | Don't restate `AGENTS.<profile>.md`; link to it |
| Concrete over abstract | *"Don't edit `generated/types.ts`"* beats *"don't edit generated code"* |
| Prefer pointers over enumeration | Volatile lists (workflow files, generated paths, env vars) ‚Üí link to the grep, header line, or directory; enumerate only the high-signal exceptions |
| Stable principles over volatile inventories | Conventions that change rarely belong here; long lists that drift weekly belong in code with a pointer from here |

## Size constraints

- Hard stop at ~100 lines.
- Sections exceeding ~10 lines move to `docs/` and get linked from `AGENTS.repo.md`.
- v0 does not need to be exhaustive. Sections left as `<!-- unknown ‚ÄĒ repo team to fill in -->` are valid signals of a documentation gap.

## Drift and freshness

The file ages out as the repo evolves. Mitigate at write time:

- **Footer the file with ref + commit** so drift-detection scripts have an anchor even in detached checkouts. Format: `<!-- Generated from <ref-or-detached>@<short-sha> on <YYYY-MM-DD>. -->`.
- **Use `<!-- verify -->`** to flag any content the agent surfaced from a scan that the engineer didn't directly confirm. The repo team is expected to remove the marker on read-through.
- **Don't restate volatile lists** (sync-owned files, generated paths, env-var names) ‚ÄĒ each one is a future drift point. Link to the source of truth.
- **Re-run the authoring flow on a cadence** (e.g. quarterly) or when the file accumulates `<!-- verify -->` markers.

## Exclusions ‚ÄĒ what does NOT belong

| Content | Belongs in |
|---|---|
| Language-wide conventions (style, formatter config) | `AGENTS.<profile>.md` |
| Org-wide PR rules, commit format, ticket linking | org or workspace `AGENTS.md` |
| Behavioural rules ‚ÄĒ "ask before destructive ops" | operator-level agent config (e.g. user-global `AGENTS.md` / `CLAUDE.md`) |
| Secrets, credentials, environment values | `.agents.env` with `op://` references (see the env-harness contract) |
| Anything true of every repo in this language or profile | `AGENTS.<profile>.md` |

## Worked example

`billing-contact-enrichment-service`. Approx 35 lines, well under the cap.

```markdown
# AGENTS.repo.md ‚ÄĒ billing-contact-enrichment-service

## 1. What this repo is (3)

Node.js worker that, given a customer account ID, returns the billing contact email used by downstream invoice delivery. Called by the account service after customer profile changes. Persists results in PostgreSQL for durable reuse; runs provider lookups via BullMQ workers.

## 2. Owner / on-call (3)

- Squad: Billing Platform
- On-call GitHub handle: `@<squad-on-call-handle>`

## 3. Sync-owned files (3)
```

Continuation:

```markdown

- `.github/workflows/deploy.yml` ‚ÄĒ sync-owned. Edit source at `<central-templates-repo>/workflow-templates/deploy.yml`.
- `.github/workflows/label-check.yml` ‚ÄĒ sync-owned. Edit source at `<central-templates-repo>/workflow-templates/label-check.yml`.

## 4. Generated files (3)

- `src/types/openai.gen.ts` ‚ÄĒ do not edit. Regenerated by `npm run gen:openai`.

## 5. Repo-specific build / test / run (3)

- BullMQ tests require Redis. `make devcontainer` starts it.
- Integration tests against MySQL take ~30s to bootstrap on first run; subsequent runs reuse the volume.
```

Continuation:

```markdown
- The provider lookup path is skipped by default in `make test`; enable it with `npm run test:provider` once the env-harness has resolved this repo's `.agents.env`.

## 6. Cross-repo dependencies (3)

- **Depends on:** Billing contact provider API; `billing-contact-db` (Terraform module).
- **Depended on by:** `account-service` (`POST /accounts/{id}/sync-billing-contact`).

## 7. Domain terms (3)

- `enrichment_execution` = audit row, not a job. Rows are immutable.
- `account_contact_state.last_outcome` = terminal status of last enrichment attempt; never partial.
- `Outcome` = FOUND / NOT_FOUND / INVALID_OUTPUT / TECHNICAL_FAILURE. Four-valued, not boolean.
```

- If a PR adds `persist-credentials: false` to a workflow file and CI is failing, the file is probably sync-owned. Check section 3 before editing further.
